Part 02 — MCP

MCP 是什麼？

讓 AI 能操作外部工具的標準介面。

把 AI 想成很聰明的新人，MCP 就是給他各種系統的帳號和權限。沒有 MCP，他只能告訴你怎麼做；有了 MCP，他能直接進去做。

互動流程

👤
You — 下指令
「幫我查 GitHub issue」
⚡
Claude Code (MCP Client)
整合工具清單送出
AI
LLM 思考決策
決定呼叫哪個工具 + 參數
⚡
Claude Code
去敲 MCP Server 的門
🔧
MCP Server (GitHub)
執行 → 回傳結果
👤
You — 看到答案
此循環可跑多次直到完成

Part 03 — Agent Skills

Agent Skills
是什麼？

預先寫好的工作指令包，AI 在對的時機自動呼叫，你也可以用 /指令 手動觸發。

Skills 就像公司的 SOP——與其每次都跟新人說一遍流程，不如把 SOP 寫好，叫他照做就好。

怎麼觸發

AI 自動

你說「我想討論報告內容」→ AI 判斷這是設計討論 → 自己呼叫 /brainstorming

手動呼叫

直接輸入 /test-driven-development → AI 照 TDD 流程走，不需要任何額外說明。

💡 就像我跟你們說想討論簡報，Claude 自己就去呼叫了頭腦風暴 Skill，沒有等我下指令。

/verification-before-completion完成前跑驗證

Skills 的技術優勢

Skills 載入的只有那個流程的 metadata，token 消耗非常少。

MCP每次對話都載入

裝越多工具 → 消耗越多 token 💸

Skills按需載入

你叫哪個它才載哪個 ✓

核心價值：讓 AI 每次都用你想要的方式做事，不需要每次重新說明。

/verification-before-completion完成前強制跑驗證
/requesting-code-review產出完整 review 報告
OpenSpec
⭐ 27k · MIT

褐地開發的規格管理層

在動手前先建立人與 AI 的共識——需求、設計、任務清單全部版本化管理。避免傳統 AI 工具依賴聊天記錄、需求漂移的問題。

opsx:propose建立 proposal + design + specs + tasks
opsx:apply執行 tasks，搭配 TDD
opsx:archive歸檔並同步 delta specs
opsx:verify驗證實作符合規格

輕量、支援 20+ AI 助手、無硬性階段限制。比 GitHub Spec Kit 輕，比 Kiro 靈活。

把規範寫給 AI 看

# 我的設定結構

CLAUDE.md # AI 的工作憲章（最高規則）

.claude/rules/
├── code-style.md # 命名、injection 規範
├── architecture.md # 分層架構、設計模式
├── api-design.md # URL 命名、Response 格式
├── config-security.md # 環境變數、安全實踐
├── database.md # Migration、查詢優化
├── performance.md # 效能考量
├── git.md # Commit 規範
├── testing.md # 強制 TDD 規則
└── workflow.md # 綠地／褐地流程

~/.claude/commands/
├── verify.md # 一鍵完整驗證
└── learn.md # Session 知識提取
這些規則寫好之後，我不需要每次 code review 都說同樣的話——AI 第一次就會照規範寫。

URL 要用 kebab-case — AI 記住了

先寫測試再實作 — AI 記住了

Response 格式統一 — AI 記住了

禁止 field injection — AI 記住了
